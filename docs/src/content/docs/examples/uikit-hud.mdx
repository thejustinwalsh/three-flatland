## What This Example Shows

This example demonstrates our fork of [`uikit`](https://github.com/pmndrs/uikit) composing directly with three-flatland's renderer — a screen-space `Fullscreen` HUD and a world-space panel share the same scene as a `TileMap2D` + `Light2D` dungeon. uikit brings a flexbox layout engine and reactive components (`Container`, `Text`, `Svg`, `Input`); our fork replaces its rendering internals so those components draw through Flatland's batched, TSL-native pipeline instead of MSDF atlases and per-draw-call meshes.

| Feature            | Demonstrated                                                                                |
| ------------------ | --------------------------------------------------------------------------------------------- |
| Screen-space HUD    | `Fullscreen` container attached to the orthographic camera, laid out with flexbox            |
| World-space panel   | A `Container` parented directly into the scene, sitting over the tilemap like a signpost      |
| Text batching       | Two `Text` components render through [`@three-flatland/slug`](../../guides/slug-text/) — analytic Bézier glyphs, ONE draw call for both |
| Icon batching       | Any number of `Svg` icons batch into ONE draw call, tinted and clipped per-instance            |
| Lit, shadowed panels | Panels are real TSL node materials — they receive `Light2D` lighting and cast shadows with correct rounded-corner silhouettes |
| Cross-backend       | The same TSL graph renders on WebGPU and WebGL2                                                |

## Why It Matters

Most game UI libraries either render to a DOM overlay (breaking immersion, fighting z-order with the canvas) or bring their own bespoke text/sprite renderer that duplicates work your 2D scene is already doing. uikit gives you a real flexbox layout engine and a component model your team already knows from the web — our fork keeps that model but retargets every pixel it draws through Flatland's existing batching and lighting machinery. A `Text` label in your HUD and a `SlugText` label floating over a sprite are the same draw call family; a `Container`'s rounded-rect background and a `Sprite2D`'s panel are the same material family. You get uikit's ergonomics without paying a second render pipeline's frame budget.

## Mounting a Screen-Space HUD

`Fullscreen` is a `Container` that sizes itself to the viewport and renders relative to the camera — pass the renderer and lay out children with ordinary flexbox properties.

<Tabs syncKey="framework">
  <TabItem label="Three.js" icon="simple-icons:threedotjs">
    ```typescript
    import { Fullscreen, Text } from '@three-flatland/uikit'

    const hud = new Fullscreen(renderer, {
      flexDirection: 'row',
      justifyContent: 'space-between',
      padding: 16,
    })
    camera.add(hud) // Fullscreen renders relative to the camera

    const hp = new Text({ text: 'HP 24/30', fontSize: 20, color: 0xffffff })
    const gold = new Text({ text: '128g', fontSize: 20, color: 0xffcc44 })
    hud.add(hp, gold)

    function animate() {
      hud.update(delta)
      renderer.render(scene, camera)
    }
    ```

  </TabItem>
  <TabItem label="React" icon="simple-icons:react">
    ```tsx
    import { extend } from '@react-three/fiber/webgpu'
    import { Fullscreen, Text } from '@three-flatland/uikit/react'

    extend({ Fullscreen })

    function Hud() {
      return (
        <Fullscreen flexDirection="row" justifyContent="space-between" padding={16}>
          <Text text="HP 24/30" fontSize={20} color={0xffffff} />
          <Text text="128g" fontSize={20} color={0xffcc44} />
        </Fullscreen>
      )
    }
    ```

  </TabItem>
</Tabs>

## A World-Space Panel

`Container` is a `Mesh` — add it to the scene like anything else. Position it in world space and it reads as a signpost or nameplate sitting inside the tilemap, not an overlay bolted on top.

<Tabs syncKey="framework">
  <TabItem label="Three.js" icon="simple-icons:threedotjs">
    ```typescript
    import { Container, Text } from '@three-flatland/uikit'

    const signpost = new Container({
      backgroundColor: 0x16191e,
      borderRadius: 8,
      padding: 12,
    })
    signpost.add(new Text({ text: 'The Sunken Crypt', fontSize: 16, color: 0xffffff }))
    signpost.position.set(0, halfExtent * 0.6, 10)
    scene.add(signpost)
    ```

  </TabItem>
  <TabItem label="React" icon="simple-icons:react">
    ```tsx
    import { extend } from '@react-three/fiber/webgpu'
    import { Container, Text } from '@three-flatland/uikit/react'

    extend({ Container })

    function Signpost() {
      return (
        <Container
          backgroundColor={0x16191e}
          borderRadius={8}
          padding={12}
          position={[0, halfExtent * 0.6, 10]}
        >
          <Text text="The Sunken Crypt" fontSize={16} color={0xffffff} />
        </Container>
      )
    }
    ```

  </TabItem>
</Tabs>

Because the panel's background is a genuine node material (not a flat-shaded quad), it responds to the scene's `Light2D`s the same way a sprite does — the torchlight in this example warms the signpost's edge, and it casts a shadow with the same rounded-corner silhouette it renders with.

## Text and Icons Batch Together

Every `Text` in a uikit tree renders through the same [Slug](../../guides/slug-text/) pipeline as `SlugText` — no MSDF atlas, glyphs stay sharp at any zoom, and the AA fringe measurably *shrinks* rather than blurs at 8× zoom. Two `Text` components with compatible fonts draw in one call; the same holds for `Svg` — any number of icons in a tree batch into a single draw call, each tinted and clipped independently.

```tsx
<Fullscreen flexDirection="column" gap={8}>
  <Text text="Inventory" fontSize={18} color={0xffffff} />
  <Text text="4 items" fontSize={14} color={0x999999} />
  {/* both Text components above: ONE draw call */}
  <Svg src="/icons/sword.svg" width={20} height={20} />
  <Svg src="/icons/shield.svg" width={20} height={20} />
  {/* both Svg icons above: ONE draw call, independent of the Text call */}
</Fullscreen>
```
