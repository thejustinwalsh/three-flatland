# Skia WASM Architecture

How GPU-accelerated 2D rendering gets from Skia C++ to the browser.

## Build Pipeline

```
                        ┌──────────────────┐
                        │   WIT Definitions │
                        │  wit/skia.wit     │  Defines the component interface
                        │  wit/gl.wit       │  GL function imports (147 fns)
                        │  wit/wgpu.wit     │  WebGPU function imports (276 fns)
                        └────────┬─────────┘
                                 │
                    wit-bindgen c │  (Step 1: generate C glue)
                                 ▼
                ┌────────────────────────────────┐
                │  src/zig/bindings/generated/    │
                │    skia_gl.h    — C types       │
                │    skia_gl.c    — C glue code   │
                │    skia_gl_component_type.o     │
                └────────────────┬───────────────┘
                                 │
                    zig build    │  (Step 2: compile to WASM)
                                 ▼
           ┌─────────────────────────────────────────┐
           │  zig-out/bin/skia-gl.wasm                │
           │  zig-out/bin/skia-wgpu.wasm              │
           └────────────────┬────────────────────────┘
                            │
               wasm-opt -Oz │  (Step 3: size optimize)
                            │
               copy to dist │  (Step 4)
                            ▼
           ┌─────────────────────────────────────────┐
           │  dist/skia-gl/skia-gl.wasm    (~3.4 MB) │
           │  dist/skia-wgpu/skia-wgpu.wasm (~2.8 MB)│
           └─────────────────────────────────────────┘
```

Run: `node scripts/build-wasm.mjs` (or `--gl-only`, `--wgpu-only`)

Prerequisites: `pnpm run skia:setup` installs Zig, wasm-tools, wit-bindgen, binaryen.

## WASM Module Structure

Each variant (GL, WGPU) is a single `.wasm` file with three import modules and one set of exports:

### Imports (what the WASM binary needs from JavaScript)

```
┌──────────────────────────────────────────────────────┐
│  Import Module: "gl" (GL variant)                    │
│  or "wgpu" (WGPU variant)                            │
│                                                      │
│  GL:   emscripten_glActiveTexture, etc. (147 fns)    │
│  WGPU: wgpuDeviceCreateBuffer, etc. (276 fns)        │
│                                                      │
│  These are the GPU API calls that Skia makes.        │
│  JavaScript must provide real implementations.        │
├──────────────────────────────────────────────────────┤
│  Import Module: "env"                                │
│                                                      │
│  Skia runtime stubs: logging, semaphores,            │
│  flattenable init. All no-ops in browser.            │
├──────────────────────────────────────────────────────┤
│  Import Module: "wasi_snapshot_preview1"             │
│                                                      │
│  Minimal WASI: fd_*, args_*, environ_*, proc_exit.   │
│  Stubs — no real filesystem.                          │
└──────────────────────────────────────────────────────┘
```

### Exports (what the WASM binary provides to JavaScript)

Two layers of exports coexist in the binary:

1. **WIT component exports** — prefixed with `exports_skia_gl_*`
   - Generated by wit-bindgen from `wit/skia.wit`
   - Handle WIT ABI (list pointers, resource handles, option types)
   - Example: `exports_skia_gl_init(config: *list_u8_t)`

2. **Direct exports** — prefixed with `skia_*`
   - Hand-written in `core.zig` (line 563+)
   - Simpler signatures (no WIT ABI overhead)
   - Example: `skia_init()`, `skia_draw_rect(x, y, w, h, paint)`

The TypeScript API uses the **direct exports** (`skia_*`).

### GL Import Name Chain

The GL function name goes through several layers. Tracing `glActiveTexture`:

```
Skia C++ (GrGLInterface)
  → emscripten_glActiveTexture()           [C function pointer]
    → emscripten_gl_shim.c wrapper         [non-inline, addressable]
      → __wasm_import_gl_ActiveTexture()   [WASM import call]
        → import_module("gl"),
          import_name("emscripten_glActiveTexture")
          → wasm-loader-gl.ts:
            emscripten_glActiveTexture(t) { gl.activeTexture(t) }
```

Key: the WASM import name includes the `emscripten_gl` prefix (set by
`__attribute__((import_name("emscripten_glActiveTexture")))` in the header).
The internal C symbol `__wasm_import_gl_ActiveTexture` is just a linker name.

### GL Handle Tables

WebGL uses JS objects (WebGLTexture, WebGLBuffer, etc.) but WASM uses
integer handles. The GL loader maintains bidirectional handle tables:

```
JavaScript (wasm-loader-gl.ts)    WASM
  WebGLTexture obj  ←──────────→  integer handle (u32)
  textures.get(3)   ←──────────→  emscripten_glBindTexture(GL_TEXTURE_2D, 3)
```

Tables: textures, buffers, framebuffers, renderbuffers, programs, shaders,
vaos, queries, samplers, syncs, uniforms.

## Data Flow: WGPU Variant

Same architecture, different GPU module:

```
  TypeScript API
    │
    │ calls skia_* exports
    ▼
  skia-wgpu.wasm
    │
    │ core.zig → skia_c_api_dawn.cpp → Skia Graphite/Dawn
    │
    │ calls wgpu* C API functions
    ▼
  wgpu_shim/emscripten_wgpu_shim.c
    │
    │ WASM imports from module "wgpu"
    ▼
  wasm-loader-wgpu.ts
    │
    │ maps C API calls to browser WebGPU
    ▼
  GPUDevice / GPUQueue / etc. (browser native)
```

The WGPU bridge must implement the full Dawn C API surface that Skia's
Graphite backend calls. Unlike GL (which is well-tested), many WGPU
functions are currently stubs and need real implementations.

## WIT (WebAssembly Interface Types)

WIT defines the component interface but is NOT used at runtime by our
TypeScript API. It serves two purposes:

1. **C code generation** — `wit-bindgen c` generates the C headers and glue
   code that the Zig build links. This creates the `exports_skia_gl_*` layer.

2. **Type generation** — `jco types` generates TypeScript types from WIT
   (output to `dist/types/`). These are reference types, not used by the
   hand-written TS wrappers.

The direct `skia_*` exports bypass WIT entirely — they're plain WASM exports
with C ABI signatures. The TS wrappers (`SkiaContext`, `SkiaPaint`, etc.)
call these directly.

### When to regenerate WIT

Run `pnpm run skia:setup` then `node scripts/build-wasm.mjs --wit-only` when:
- The WIT files (`wit/*.wit`) change
- Adding new exported functions to the WIT interface
- Updating the Skia pin (which may add new GL/WGPU functions)

The generated files in `src/zig/bindings/generated/` are checked in.

## Context Initialization

### GL: `skia_init()`

```
skia_init()
  → sk_context_create(0, 0)        [skia_c_api_gl.cpp]
    → GrGLInterfaces::MakeWebGL()  [Skia: assembles GL function table]
      → looks up emscripten_gl* function pointers
    → GrDirectContexts::MakeGL()   [Skia: creates GPU context]
```

`MakeWebGL()` calls each `emscripten_gl*` function pointer to build
Skia's internal GL interface table. If any critical function returns
null or fails, `MakeWebGL()` returns null → init error code 1.

**Debug**: `skia_debug_init_error()` returns:
- 0: success
- 1: `MakeWebGL()` returned null (GL interface assembly failed)
- 2: `MakeGL()` returned null (context creation failed)

### WGPU: `skia_init_with_handles(device, queue)`

```
skia_init_with_handles(deviceH, queueH)
  → sk_context_create(deviceH, queueH)  [skia_c_api_dawn.cpp]
    → uses Dawn C API to wrap the handles
    → creates Graphite context
```

## Surface & Drawing

```
skia_begin_drawing(fbo_id, width, height)
  → sk_surface_create_for_target(ctx, fbo_id, w, h)
    GL:   wraps an existing FBO as a Skia render target
    WGPU: wraps a registered GPUTexture
  → sk_surface_get_canvas(surface)
    returns the Skia canvas for drawing

[draw calls: skia_draw_rect, skia_draw_circle, etc.]
  → each calls the C API which calls SkCanvas methods

skia_end_drawing()
  → sk_surface_flush(surface)
    flushes GPU commands

skia_flush()
  → sk_context_flush(ctx)
    flushes the GPU context

skia_reset_state()  [GL only]
  → sk_context_reset_state(ctx)
    resets GL state cache so Three.js can use the same context
```
